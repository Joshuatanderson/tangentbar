Tangent v2 — the OS-level tangent layer

Architecture for the new app: a macOS menu-bar agent that lets you double-click a word in any application to open a grounded definition tangent, and grab any selection into a fresh chat. v1 stays untouched as the reference implementation.

What v1 already settled — and what v2 replaces

v1 proved the interaction grammar, and its core was deliberately built UI-agnostic. That means v2 is not a rewrite of the whole app — it's a new sensing layer (where do events come from) and a new surfacing layer (where do results appear) around a brain that already exists.

Worth being explicit about the cost: v1 was cross-platform by construction; v2's entire value proposition rests on the macOS Accessibility APIAccessibility API (AX) — the macOS tree of UI elements (AXUIElement) that every well-behaved app exposes so screen readers can inspect it. It's also the only sanctioned way for one app to read text out of another app — which is exactly what v2 needs.. A Windows port later would be a parallel sensing/surfacing layer over UIA, not a code share.

Everything the new app does decomposes into four layers. Architectural choice — the popup is a non-activating panel, not a context menuWhat you described ("the window you get when you right-click") is technically an NSMenu, but menus can't host streaming text or an input field. The right primitive is a non-activating NSPanelNon-activating NSPanel — a floating window with the .nonactivatingPanel style: it appears above everything (level .popUpMenu) and accepts clicks without taking focus away from the app you're reading. Spotlight and the Look Up popover are built this way. anchored at the mouse point. Critical property: focus never leaves the app you double-clicked in — you keep reading, the tangent floats. Esc or click-away dismisses; a small "open as chat ↗" affordance promotes it to a real window.
Architectural choice — Swift owns senses & surfaces, Rust owns thinking & memoryThe event tap, AX reads, TCC prompts, and panel behavior are the flakiest, most platform-idiomatic 10% of the app — first-class and heavily documented in Swift, rough and thinly-wrapped in Rust crates. Meanwhile the transports/history/keys are the most valuable code you have, and they're Rust. The split puts each half where it's strong. Decision 1 lets you overrule this.

Event flow A — the system-wide tangent (double-click)

Sensor
⌥-double-click fires in Safari (trigger per Decision 2). The CGEventTapCGEventTap — a low-level hook on the system event stream. A listen-only tap sees every mouse event OS-wide before/as apps get them; a mouse-only tap needs the Accessibility permission (keyboard taps additionally need Input Monitoring — v2 avoids those by using the Carbon hotkey API instead). sees leftMouseUp with clickCount == 2 + ⌥ held. Events inside Tangent's own windows are ignored.
Extraction
AXUIElementCopyElementAtPosition(point) → the text element under the cursor. Read the double-click-selected word (kAXSelectedTextAttribute) and pull ~2 sentences each side via parameterized AX attributesParameterized AX attributes — AX getters that take an argument, e.g. kAXRangeForPositionParameterizedAttribute (character range at a screen point) and kAXStringForRangeParameterizedAttribute (the text in a range). Together they let you read "the word here, plus N characters around it" from another app.
Fallback ladder below if the app's AX support is poor.
Surface
Popup panel appears at the cursor immediately (<100 ms budget) showing the word + a typing indicator. Focus stays in Safari.
Brain
Shell writes {"op":"tangent","word":…,"context":…,"source_app":"Safari"} to tangentd; the daemon runs resolve_model (explicit pick → Haiku → local) and streams chunks back; the panel renders them as they arrive.
Surface
Esc / click-away → panel closes, tangent dropped (v1 semantics: disposable). Typing a follow-up keeps it alive; "open as chat ↗" promotes it to a full window and hands the history to the explore arena.

Event flow B — grab a selection into a chat

Sensor
You've drag-selected a paragraph anywhere. Trigger fires (hotkey vs. pill — Decision 3).
Extraction
Read the focused app's focused element (kAXFocusedUIElementAttribute) → kAXSelectedTextAttribute, plus surrounding context via the range attributes. Record source app + window title as provenance.
Surface + Brain
New chat window opens seeded with the excerpt — exactly a v1 explore Node with parent: None, title = snippet, context = selection + margins. It waits for your first message (v1's manual-send semantics), full model picker available.

The fallback ladder (per trigger, in order)

  • 1. Full AX — word + ±context via range attributes. The good path.
  • 2. Selection-only AX — kAXSelectedTextAttribute works but ranges don't: you get the word/selection, no surrounding context. Tangent still works, just less grounded.
  • 3. Clipboard synthesis — save the pasteboard, synthesize ⌘C, read, restore. Gets whatever the app considers selected. Last resort: it's observable (pasteboard managers see it) and racy; gate it behind a setting, default on for known-poor apps only.
  • 4. Give up loudly — brief "nothing readable here" hint in the popup rather than silent failure, so trust in the trigger stays intact.
Needs your review — the noise problemA bare, unmodified global double-click is extremely high-frequency: you double-click to select-and-retype text constantly while editing. Popping a panel on every one would make you uninstall your own app in a day. That's why Decision 2 exists and why my recommendation adds a modifier. The prior art agrees: macOS Look Up is ⌃⌘D or a deliberate three-finger tap; PopClip shows only a tiny pill; Raycast selection actions are hotkey-invoked.

Permissions, signing, lifecycle

  • Accessibility (TCC) — one grant covers both the mouse event tap and all AX reads. Prompt once on first launch via AXIsProcessTrustedWithOptions with a friendly explainer window; poll until granted, then arm the tap. TCCTCC — Transparency, Consent & Control: the macOS per-app permission database behind System Settings → Privacy & Security. Grants are keyed to the app's bundle identity and code signature.
  • Avoid Input Monitoring entirely — keyboard event taps need a second, scarier permission. Use the Carbon RegisterEventHotKey API for the grab hotkey (no permission needed) and keep the tap mouse-only.
  • Sign with a stable identity from day one. TCC grants are tied to the code signature — an ad-hoc signature that changes every build silently revokes the grant, and "why did my tap stop working" becomes a daily mystery. Use your Developer ID (or a consistent self-signed cert) even for dev builds.
  • No App Sandbox → no Mac App Store. AX reads of other apps are fundamentally incompatible with sandboxing. Distribution is a notarized DMG/zip, like PopClip, Raycast, Rectangle.
  • Menu-bar agent — LSUIElement = true (no Dock icon, no app switcher entry); launch at login via SMAppService. The status-item menu holds: enable/disable toggle, per-app exclusion list, tangent-model and chat-model pickers, history window, settings, quit.
Hard constraintThe event tap dies if the process is slow to service it (the OS disables unresponsive taps). The tap callback must do nothing but filter and enqueue — all AX reads and daemon I/O happen off that thread, and the tap gets re-enabled on kCGEventTapDisabledByTimeout.
AssumptionLatency budget: popup visible <100 ms, first streamed token <1 s. This drives three choices — the popup surface is pre-warmed at launch, tangentd stays resident (not spawned per request), and the tangent default model stays cheap (Haiku or a local model), exactly as v1's resolve_model already encodes.

    Surface/ChatWindow.swift     #   promoted chats
    Brain/DaemonClient.swift     #   spawn tangentd, JSON-lines framing, streams
  core/                          # Rust workspace
    tangent-core/                #   extracted v1 lib: model, tangent, explore,
                                 #   transports, history, persist (no UI deps)
    tangentd/                    #   thin bin: JSON-lines stdio loop over the core
  protocol.md                    # the stdio contract, versioned
  assets/                        # v1 brand carries over (icon, ink palette CSS)

The protocol is deliberately boring — one JSON object per line: requests down (tangent, grab, followup, set_model, list_models), streamed events up (chunk, done, error, keyed by request id). It's the same shape tangent-cli already implies, and it keeps the Swift/Rust boundary testable from the command line with nothing but echo | tangentd.
