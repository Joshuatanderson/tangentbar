Contextual definitions — where context comes from

Architecture review · no implementation yet · TangentBar, 2026-07-03

TL;DR

The engine already defines contextually on the good extraction rungs: rung 1 and 2 capture ±400 characters (≈60–70 words) around the clicked word via the AX APIAX (Accessibility) API — macOS's assistive-technology interface. Apps expose their text and UI structure through it, so we can read the text around a click without touching the user's selection or clipboard., and the prompt already carries it as a Passage:. That's more than the 10–20 words each side you asked for.

Why it felt non-contextual Your test ran in cmux, which extracts via rung 3a-copy-on-select — the pasteboard contains only the double-clicked word, so context == word and the model defines blind. The gap is specific to the clipboard rungs (GPU terminals, and anything else where AX fails), not global. The architecture question is: how do we get surrounding text where AX can't give it to us?
Architectural clarification: no visible highlight needed "Highlight more than one word" doesn't require selecting more text in the target app. On AX rungs the context is read invisibly (AXStringForRange) — the user's one-word selection stays untouched. Actually expanding the visible selection would mean mutating another app's state, and is rejected below. If you specifically want the passage visibly highlighted as feedback, that's a different (UI) feature — say so in notes.

Baseline fix regardless of anything below: when context == word, the prompt should say "no surrounding context available" instead of presenting the bare word as a passage — the model stops pretending and gives a clean general definition.

Options for the gap

A — Parent/sibling AX gathering (fixes the "clamped" rows)

When the hit element's text is tiny, read the parent's other AXStaticText children (bounded: one or two levels, N siblings) and stitch a passage. Already a listed work item from the spike.
Fidelity: exact text · Coverage: chat apps & fragmented UIs · Permissions: none new · Latency: ~ms · Risk: low (read-only AX)

B — Terminal adapter (fixes cmux/ghostty — your daily driver)

Ask the terminal itself for its buffer: tmux capture-pane -p when the pane runs tmux, or the cmux CLI's pane-text read. Locate the copied word in the dump, window ±N words. Exact text, no OCR noise, no new permissions.
Fidelity: exact text · Coverage: terminals with a CLI (cmux/tmux; plain ghostty has no dump CLI) · Permissions: none · Latency: ~10–50 ms · Risk: coupling to CLI availability
Assumption to verify The double-click that triggered us also focused the clicked pane, so "active pane" == "clicked pane" when we shell out. Believed true (first click focuses), needs a real-world check — and multi-session targeting via $TMUX-less lookup needs a rule.

C — OCR rung (the universal floor — was already penciled as "v3")

Screenshot a region around the click (ScreenCaptureKitScreenCaptureKit — Apple's modern screen-capture framework. Any use of it requires the user to grant Screen Recording permission to the app.) and run Apple's on-device Vision text recognizer. Bounding boxes let us locate the clicked word geometrically and assemble the passage in reading order. Works everywhere: bare ghostty, Zed, Google Docs canvas, scanned PDFs, images, video frames.
Fidelity: OCR-grade (errors, column/layout garbling) · Coverage: universal · Permissions: Screen Recording · Latency: ~100–300 ms · Risk: moderate complexity
The permission is the real cost Screen Recording is a heavy TCCTCC — Transparency, Consent, and Control: macOS's per-app permission database (the Privacy & Security prompts). Screen Recording is one of its scariest grants, and macOS Sequoia re-nags for it periodically. grant, and recent macOS re-prompts for it periodically. For a personal tool that's a one-time annoyance; for anything distributed it's a real adoption tax. It also philosophically changes the app from "reads accessibility text" to "can see your screen".

D — Synthetic selection expansion — rejected

Synthesize ⇧⌥→/triple-click to grow the real selection, copy, restore. Visibly stomps the user's selection, racy to restore, per-app divergent behavior, and terminals interpret those keys as input. The clipboard rung (3b) is already the most invasive thing we do; this doubles down in the wrong direction.
Fidelity: exact · Coverage: broad in theory · Risk: user-visible side effects — disqualifying
Shape: a context ladder mirroring the word ladder These aren't alternatives — they compose, ordered by fidelity: AX range → AX value → sibling gathering (A) → app adapter (B) → OCR (C). Each rung fires only when the one above produced fewer than ~N words of context. One new concept in the codebase, not three features.
